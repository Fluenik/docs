---
title: Approval flows
description: "When a policy says 'require approval,' a human has to say yes. BRAM surfaces that request where your team already works."
---

`require_approval` is the middle ground between `allow` and `deny`. The tool call doesn't execute until a human confirms it, but it also doesn't just die — the decision is captured and the call proceeds (or is cancelled) based on the human's response.

## Surfaces

BRAM ships **three approval surfaces**, any combination of which can be active at once:

<CardGroup cols={3}>
  <Card title="Native OS modal" icon="desktop">
    BRAM Desktop surfaces a native modal (macOS UNUserNotification, Windows toast, Linux libnotify). Full tool context, one click to approve / deny.
  </Card>
  <Card title="Slack" icon="slack">
    Post the approval request as an interactive message in a channel or DM. Approve / deny buttons are wired back to the daemon.
  </Card>
  <Card title="Webhook" icon="webhook">
    POST the full approval payload to a URL of your choice; call the callback URL to approve or deny. Use for custom dashboards or ticketing integration.
  </Card>
</CardGroup>

## What a request looks like

When a policy fires `require_approval`, the approval surface gets:

```json
{
  "approval_id": "apr_01HK…",
  "tool": "github.create_issue",
  "arguments": {
    "owner": "acme",
    "repo": "billing",
    "title": "Customer #4271 reports double-charge",
    "body": "The customer says…"
  },
  "agent_context": {
    "client": "cursor",
    "conversation_id": "conv_…",
    "recent_prompts": ["...", "..."]
  },
  "policy": {
    "rule_index": 3,
    "reason": "GitHub write: github.create_issue"
  },
  "expires_at": "2026-04-23T19:12:00Z"
}
```

`agent_context` includes the last few prompts (redacted if you enable that) so a reviewer can see **why** the agent decided to make this call.

## Responding

Approvals are open-ended — "approve once" is default, but reviewers can also choose:

| Response | Scope |
|---|---|
| **Approve once** | This exact call only. |
| **Approve for 10 min** | Any identical (`tool` + normalized args) call in a 10-minute window. |
| **Approve for the session** | Until BRAM Desktop restarts. |
| **Deny** | Block this call; agent sees a tool error. |
| **Deny + block rule** | Block this call **and** add a deny rule to the policy so future calls short-circuit. |

Approval scope is captured in the audit log alongside the decision.

## Configure approval surfaces

### Native (local only)

On by default when BRAM Desktop is running. No config needed.

### Slack

<Steps>
  <Step title="Install the BRAM Slack app">
    From the dashboard: **Settings → Integrations → Slack → Install**.
  </Step>
  <Step title="Pick the channel">
    Approvals post to the channel you pick. DMs work too — for single-reviewer workflows.
  </Step>
  <Step title="Tell BRAM who can approve">
    In the policy, set `approval.slack_approvers` to a list of Slack user IDs or group IDs. Anyone else clicking the button gets a "not authorised" response.
  </Step>
</Steps>

```yaml
# in policy.yaml
- match:
    tool: "gmail.send_email"
  decision: require_approval
  approval:
    surfaces: ["slack", "native"]
    slack_channel: "C01ABC2DEF"
    slack_approvers: ["U01XY12AB", "G01OWNERS"]
    timeout: "5m"               # auto-deny after 5 minutes with no response
```

### Webhook

Point at any HTTPS endpoint. BRAM POSTs the approval payload; your service calls a callback URL to respond.

```yaml
- match:
    tool: "database.execute"
    args:
      database: "prod"
  decision: require_approval
  approval:
    surfaces: ["webhook"]
    webhook_url: "https://my-app.example.com/bram/approvals"
    webhook_secret: "${BRAM_APPROVAL_HMAC}"
    timeout: "10m"
```

Webhook signature verification uses HMAC SHA256 with `webhook_secret`; the signature is in the `X-BRAM-Signature` header.

Your handler calls:

```bash
curl -X POST \
  -H "Authorization: Bearer $BRAM_RUNTIME_KEY" \
  "https://api.getbram.com/api/governor/v1/approvals/$APPROVAL_ID/decide/" \
  -d '{"decision": "approve", "scope": "once", "actor": "alice@acme.com"}'
```

## Timeouts + escalation

Every approval has a `timeout`. If no one responds within the window, the default is to **deny** — this is the safe default, but you can override per-rule:

```yaml
approval:
  timeout: "10m"
  on_timeout: "deny"        # or "approve" (rare) or "escalate"
  escalate_to:              # only if on_timeout is "escalate"
    surfaces: ["slack"]
    slack_channel: "C01ONCALL"
```

Escalation re-issues the approval request to a different surface — typically a louder one (pager, oncall channel).

## Auditing

Every approval (including timeouts) lands in the audit log with:

- Request payload (tool, args, agent context)
- Time request was sent, time responded
- Responder identity (OS user, Slack user, or webhook signer)
- Scope chosen (once / 10m / session)
- Resulting tool call outcome

See [Audit logs](/governance/audit-logs).
