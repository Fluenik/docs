BRAM policies are versioned rule bundles managed in the dashboard at [app.getbram.com](https://app.getbram.com) under **Tools & Policies**. Each policy is a list of **rules**, and each rule matches on tool name, arguments, or context and produces a **decision**: `allow`, `deny`, or `require_approval`.

You can edit policies in the dashboard UI, or POST the same YAML/JSON via the [Governor API](/api-reference/authentication). Both paths produce versioned, auditable changes — every edit creates a new immutable version with `draft` / `canary` / `live` states and one-click rollback.

## Policy structure

```yaml
version: 1
description: "Default policy — auto-approve reads, prompt for writes, deny destructive ops."

rules:
  # 1. Auto-approve safe read operations
  - match:
      tool: "filesystem.read_file"
    decision: allow

  # 2. Prompt for anything that writes to disk
  - match:
      tool: "filesystem.write_file"
    decision: require_approval
    reason: "Writing to {{args.path}}"

  # 3. Block writes outside an allowed prefix
  - match:
      tool: "filesystem.write_file"
      args:
        path: "!startswith:/workspace/projects/"
    decision: deny
    reason: "Writes outside projects/ are not permitted"

  # 4. Require approval for any GitHub write
  - match:
      tool_prefix: "github."
      args:
        # any mutation — create, update, merge, delete
        $action: "!=read"
    decision: require_approval
    reason: "GitHub write: {{tool}}"

  # 5. Default — fall through to approval
  - decision: require_approval
    reason: "Unclassified tool call"
```

Rules are evaluated **top-down**. The first match wins. The final rule without a `match` acts as a catch-all.

## Match predicates

### By tool name

```yaml
match:
  tool: "gmail.send_email"               # exact match
  tool_prefix: "shopify."                # prefix match
  tool_glob: "github.create_*"           # glob match
```

### By argument

```yaml
match:
  tool: "database.execute"
  args:
    statement: "regex:^(DROP|TRUNCATE|DELETE)"    # regex on a string arg
    database: "prod"                               # exact match
    row_limit: ">1000"                             # numeric comparison
    dry_run: false                                 # boolean
```

Supported operators:

| Operator | Example | Matches |
|---|---|---|
| *(none)* | `name: foo` | equals |
| `regex:` | `statement: "regex:^DROP"` | regex match |
| `glob:` | `path: "glob:/tmp/*"` | glob match |
| `startswith:` | `path: "startswith:/etc"` | prefix match |
| `!startswith:` | `path: "!startswith:/tmp"` | negated prefix |
| `in:` | `domain: "in:prod,staging"` | membership |
| `>`, `>=`, `<`, `<=` | `amount: ">1000"` | numeric comparison |

### By context

```yaml
match:
  tool: "slack.post_message"
  context:
    agent_id: "support-bot-prod"
    time: "outside:09:00-17:00"        # business hours
    day: "in:sat,sun"                  # weekends
```

`context.agent_id` lets a single policy apply different rules per agent — useful for one shared policy across many agents.

Every decision is recorded to the audit log with the matching rule's index and the policy version, so you can trace exactly why a call was allowed or blocked.

## Time-boxed approvals

Approvals can be **remembered for a window**:

```yaml
- match:
    tool_prefix: "gmail."
  decision: require_approval
  remember:
    scope: "agent"            # or "workspace"
    duration: "10m"           # or "1h", "session"
    key: "gmail-all"          # grouped so one approval covers all gmail.*
```

The approval card will show a **Remember for 10 min** option. Pick it, and identical calls in the window auto-approve without re-prompting.

## Versioning + rollouts

Every save creates a new immutable policy version. Versions move through states:

- **draft** — editable, not enforced
- **canary** — enforced for a fraction of traffic (default 10%) so you can compare decisions before promoting
- **live** — enforced for all traffic

Rollouts are managed in the dashboard under **Tools & Policies → Versions**. One-click rollback restores the previous live version instantly.

## Testing policies

Run a policy against a fixture corpus from the dashboard or via API:

```bash
curl -X POST https://api.getbram.com/api/governor/v1/policies/$POLICY_ID/test/ \
  -H "Authorization: Bearer $BRAM_API_KEY" \
  -H "Content-Type: application/json" \
  -d @fixtures.json
```

`fixtures.json` is a list of `{ tool, args, expected_decision }`. The response reports per-fixture matches and mismatches. CI can use this to gate policy changes before they're promoted from canary to live.
