## The problem

Modern AI agents don't just chat — they send emails, write to your database, move money, and touch production systems. Most MCP clients auto-approve tool calls or ask once and never again. That "ask once" model breaks down the second your agent runs unattended, and it doesn't help at all when you have many agents in production.

**BRAM Gateway** is a hosted MCP gateway that sits between your agents and every MCP server they call. It enforces policy at the tool-call boundary, runs inline guardrails on requests and responses, and writes a tamper-evident audit log of every decision.

## Architecture

```
┌─────────────┐         ┌──────────────┐         ┌───────────────┐
│ Your agent  │ ─MCP──▶ │ BRAM Gateway │ ─MCP──▶ │  MCP servers  │
│  (Claude,   │         │  policy +    │         │ (GitHub, DB,  │
│   Cursor,   │ ◀─tool─ │  guardrails  │ ◀─tool─ │  Slack, etc.) │
│   custom)   │  result │  + audit     │  result │               │
└─────────────┘         └──────────────┘         └───────────────┘
                               │
                               ▼
                        ┌──────────────┐
                        │ Audit log +  │
                        │ trace export │
                        └──────────────┘
```

Your agent connects to the BRAM Gateway endpoint exactly the way it would connect to any MCP server. The gateway speaks native MCP, applies your policies, and forwards allowed calls upstream to the real MCP servers you've registered. Decisions are: **allow**, **deny**, or **require human approval**.

## What it does

<CardGroup cols={2}>
  <Card title="Policy enforcement" icon="shield-check">
    Versioned rules evaluated at the tool-call boundary — `gmail.send_email` requires approval, `shopify.update_order` is blocked outside business hours, `filesystem.write` is denied for paths outside an allowed prefix.
  </Card>
  <Card title="Inline guardrails" icon="shield">
    Pattern-based detectors run on every gateway request and response — prompt-injection on the way in, data-leakage on the way out. Findings can redact the payload, escalate, or block.
  </Card>
  <Card title="Approval workflows" icon="bell">
    When a policy says "require approval," BRAM routes the request to your dashboard, Slack, or a webhook. The tool call pauses until a human decides.
  </Card>
  <Card title="Searchable audit log" icon="magnifying-glass">
    Every tool call, every policy decision, every approval — captured with timestamps, arguments, results, and a signed chain of custody.
  </Card>
</CardGroup>

## Why a gateway

BRAM Gateway runs in the cloud, with one shared control plane for every agent in your workspace. That matters because:

- **No code changes** — agents point at a URL instead of a local server. No SDK, no daemon, no install.
- **One policy plane** — policies, guardrails, and audit live with your team, not on each developer's laptop.
- **Production traffic** — designed for many agents, many MCP servers, and serious request volume. Per-workspace quotas, per-agent kill switch, canary policy rollouts.
- **Deployment options** — managed cloud on Starter and Pro; on-prem or VPC on Enterprise so traffic never leaves your perimeter.
